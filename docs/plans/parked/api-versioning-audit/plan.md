# API Versioning & Schema Migration — Codebase Audit

Context document for evaluating whether to add an umbrella API versioning/capability
layer. Written as a reference for an external reviewer or AI agent.

## Question

Should this codebase introduce a formal API versioning / capability negotiation layer
now — or is the current ad-hoc approach sufficient? The project may expand to mobile
clients, plugin SDKs, or CLI tools in the future.

## Current State

The codebase has **fragmented, per-domain versioning** with no unified contract.

## Gaps

1. **No schema version in REST payloads** — the backend cannot distinguish a v1 preference
   dict from a v2 one. Canonicalizers infer shape by checking which keys exist, which is
   fragile as shapes diverge.

2. **No client capability negotiation** — the frontend hardcodes `/api/v1` and assumes all
   endpoints exist. If the backend drops or changes an endpoint, the client gets a raw 404/422.

3. **No deprecation signaling** — no `Sunset` or `X-Deprecated` headers. Old clients won't
   know they're using a stale API.

4. **Discovery endpoint is dev-only** — `/dev/info` is guarded and returns static strings, not
   machine-readable capability declarations with versions.

5. **Per-store migration is ad-hoc** — each zustand store invents its own migration logic.
   No shared `createVersionedStore()` helper.

## Options to Evaluate

### A. Do Nothing (status quo)
Keep per-domain canonicalizers and per-store migrations. Accept the cost of ad-hoc
inference when shapes change.

### B. Lightweight: Schema Version Tags + Shared Store Migration Helper
- Add `_v` field to backend preference/config payloads.
- Backend canonicalizers dispatch on `_v` instead of key-sniffing.
- Frontend: shared `createVersionedPersist(migrations)` wrapper for zustand stores.
- No new endpoints; no protocol changes.

### C. Medium: Capability Discovery Endpoint
- New `GET /api/meta` returning `{ capabilities, schema_versions, deprecations }`.
- Frontend checks on startup, adapts or shows upgrade notice.
- Backend adds `Sunset` header on deprecated endpoints.

### D. Full: Content Negotiation + Multi-Version Routers
- Support `/api/v1/` and `/api/v2/` simultaneously.
- `Accept: application/vnd.pixsim.v2+json` header support.
- Consumer-driven contract testing.
